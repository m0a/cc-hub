# Feature Specification: CC Hub TUI

**Feature Branch**: `002-cchub-tui`

**Created**: 2026-05-31

**Status**: Draft

**Input**: User description: "CC Hub の TUI 版。CC Hub サーバが動作しているローカル環境で動かす tmux ラッパー的なターミナル UI。メイン=アクティブセッション一覧表示、サブ=セッション履歴検索。端末はネットワーク転送せずローカルの tmux attach で完結。他ピアは対象外。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - セッション一覧の閲覧と入室 (Priority: P1)

開発者として、ターミナルから CC Hub のアクティブなセッション一覧をひと目で把握し、目的のセッションに素早く入って作業を続けたい。ブラウザを開かずとも、いま動いているマシン上で完結させたい。

各セッションについて、稼働中のエージェント種別(Claude Code / Codex)、現在の状態(処理中 / 入力待ち / アイドル / 完了)、作業ディレクトリ、付与タイトル、ペイン数が分かる。一覧から1つ選んで入室すると、そのセッションのターミナルをネイティブの操作感(マウス・スクロール・コピー等すべて)で扱える。作業を終えて離脱すると、また一覧に戻る。

**Why this priority**: これがこの TUI の核心。一覧表示と入室がなければツールとして成り立たない。これ単体で「ローカルのセッションを見て・入る」という価値が完結する MVP になる。

**Independent Test**: CC Hub サーバが稼働している状態で TUI を起動し、現在のセッションが状態付きで一覧表示されること、任意のセッションを選んで入室・離脱して一覧へ戻れることを確認する。

**Acceptance Scenarios**:

1. **Given** ローカルで複数のセッションが稼働している, **When** TUI を起動する, **Then** 各セッションがエージェント種別・状態・作業ディレクトリ・タイトル・ペイン数とともに一覧表示される
2. **Given** あるセッションが「処理中」から「入力待ち」へ変化した, **When** 一覧を表示し続けている, **Then** 数秒以内にそのセッションの状態表示が更新される
3. **Given** セッション一覧が表示されている, **When** 1つを選んで入室操作を行う, **Then** そのセッションのターミナルに切り替わり、通常どおり操作できる
4. **Given** セッションに入室している, **When** 離脱操作を行う, **Then** TUI のセッション一覧に戻り、一覧は最新状態で再描画される
5. **Given** 稼働中のセッションが1つも無い, **When** TUI を起動する, **Then** 「セッションがありません」と分かる空状態が表示される

---

### User Story 2 - セッション履歴の検索と再開 (Priority: P2)

開発者として、過去に行った Claude Code / Codex のセッションを横断的に検索し、見つかった会話からその作業を再開(resume)したい。「あの作業どこでやったっけ」を素早く引き当てたい。

**Why this priority**: 一覧(現在)に対して、履歴(過去)へのアクセスを提供する2つ目の柱。一覧機能とは独立して動作・検証でき、単体でも「過去作業の発見と再開」という価値を提供する。

**Independent Test**: 検索ビューでキーワードを入力し、過去セッションが逐次表示されること、結果の1件を選んで再開すると当該セッションがアクティブになり入室できることを確認する。

**Acceptance Scenarios**:

1. **Given** 過去のセッション履歴が存在する, **When** 検索ビューでキーワードを入力する, **Then** 一致するセッションが逐次(入力に追従して)表示される
2. **Given** 検索結果が表示されている, **When** 1件を選んで再開操作を行う, **Then** そのセッションが再開され、入室できる状態になる
3. **Given** 一致する履歴が無いキーワードを入力した, **When** 検索する, **Then** 「該当なし」と分かる空状態が表示される

---

### User Story 3 - セッションのライフサイクル操作 (Priority: P3)

開発者として、一覧画面から新しいセッションを作成したり、不要になったセッションを終了したり、停止中のセッションを resume したりしたい。一覧から離れずに基本操作を完結させたい。

**Why this priority**: 閲覧・入室(P1)と履歴(P2)が揃った上での利便性向上。無くても P1/P2 は成立するが、あると一覧画面が操作の起点として完成する。

**Independent Test**: 一覧画面から新規作成(エージェント種別と作業ディレクトリを指定)・終了・resume をそれぞれ実行し、操作後に一覧へ反映されることを確認する。

**Acceptance Scenarios**:

1. **Given** 一覧が表示されている, **When** 新規セッション作成でエージェント種別と作業ディレクトリを指定して実行する, **Then** 新しいセッションが作成され一覧に現れる
2. **Given** あるセッションを選択している, **When** 終了操作を行う, **Then** そのセッションが終了し一覧から消える
3. **Given** resume 可能なセッションを選択している, **When** resume 操作を行う, **Then** セッションが再開され入室できる

---

### Edge Cases

- **CC Hub サーバが起動していない**: 接続できない旨と、サーバを起動する手順を分かりやすく提示する(無言のハングや不可解なエラーで終わらない)。
- **TUI 自身が tmux セッション内から起動された(ネスト)**: そのままの入室方式では端末アタッチが拒否されるため、ネストを検知して破綻せずに入室を成立させる。
- **入室中に対象セッションが外部要因で消滅(別経路で kill 等)**: 一覧へ安全に戻り、消滅を反映する。
- **認証が必要な構成(サーバにパスワードが設定済み)**: ローカル利用では追加の手入力を求めず利用できる。
- **履歴検索の結果が大量 / ゼロ**: 大量時も操作可能な範囲で表示し、ゼロ時は空状態を示す。
- **一覧の状態が頻繁に変化**: 過剰なちらつきや負荷なく更新が反映される。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、ローカルで現在稼働中のセッション一覧を表示しなければならない。
- **FR-002**: システムは、各セッションについて「エージェント種別(Claude Code / Codex)」「状態(処理中 / 入力待ち / アイドル / 完了)」「作業ディレクトリ」「付与タイトル」「ペイン数」を表示しなければならない。
- **FR-003**: システムは、セッションの状態変化を継続的に取り込み、一覧表示へ反映しなければならない(ユーザーが手動更新せずとも更新される)。
- **FR-004**: ユーザーは、一覧から1つのセッションを選んで入室でき、入室後はそのセッションのターミナルをネイティブと同等の操作性(スクロール・選択・コピー等)で扱えなければならない。
- **FR-005**: ユーザーが入室から離脱すると、システムは TUI のセッション一覧へ戻り、一覧を最新状態で再描画しなければならない。
- **FR-006**: システムは、入室・表示の対象をローカルのセッションに限定し、他ピアで稼働するセッションは対象外としなければならない。
- **FR-007**: ユーザーは、過去のセッション履歴をキーワードで横断検索でき、結果は入力に追従して逐次表示されなければならない。
- **FR-008**: ユーザーは、検索結果のセッションを再開(resume)でき、再開後そのセッションへ入室できなければならない。
- **FR-009**: ユーザーは、一覧画面からエージェント種別と作業ディレクトリを指定して新しいセッションを作成できなければならない。
- **FR-010**: ユーザーは、一覧画面からセッションを終了できなければならない。
- **FR-011**: ユーザーは、一覧画面から resume 可能なセッションを再開できなければならない。
- **FR-012**: システムは、サーバへ接続できない場合に、原因(サーバ未起動など)と対処手順を分かりやすく提示しなければならない。
- **FR-013**: システムは、TUI がネストした端末(既存の tmux セッション内)から起動された場合でも、入室操作を破綻させず成立させなければならない。
- **FR-014**: システムは、認証が必要な構成であっても、ローカル利用において追加の手入力なしで利用できなければならない。
- **FR-015**: システムは、稼働中セッションが存在しない場合・検索結果が存在しない場合に、エラーではなく明確な空状態を提示しなければならない。

### Key Entities *(include if feature involves data)*

- **セッション (Session)**: 現在稼働中の作業単位。属性として識別子・表示名・エージェント種別・状態インジケータ・作業ディレクトリ・付与タイトル・ペイン群を持つ。
- **状態インジケータ (Indicator State)**: セッションの現在状態。`処理中 / 入力待ち / アイドル / 完了` のいずれか。
- **セッション履歴エントリ (Session History Entry)**: 過去のセッションを表す。属性としてプロジェクト(作業ディレクトリ)・要約/会話の手がかり・最終アクセス時刻・再開のための識別子を持つ。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: TUI 起動から3秒以内に、現在のセッション一覧(状態付き)が表示される。
- **SC-002**: セッションの状態変化が、一覧表示へ5秒以内に反映される。
- **SC-003**: 一覧上の目的セッションを、2回以下のキー操作で選択・入室できる。
- **SC-004**: 履歴検索でキーワードを入力後、最初の結果が2秒以内に表示され始める。
- **SC-005**: 「入室 → 離脱 → 一覧復帰」を、セッションの内容(スクロールバック・実行中の状態)を失わずに何度でも往復できる。
- **SC-006**: CC Hub サーバが稼働しているローカル環境で、追加設定ゼロで(インストール後すぐに)利用を開始できる。
- **SC-007**: ネストした端末から起動した場合でも、入室が成功する(失敗・ハングしない)。

## Assumptions

- **稼働前提**: 同一マシンで CC Hub サーバが稼働していることを前提とする。TUI はそのサーバが集約済みのセッション情報・状態・履歴をデータ源として利用し、独自に再集約しない(状態インジケータの即時性はサーバが提供する範囲に依存する)。
- **スコープ境界**: 他ピアで稼働するセッションの表示・操作は v1 のスコープ外(ローカルのみ)。
- **端末方式**: ターミナル画面をネットワーク越しに転送しない。入室はローカルのネイティブ端末アタッチで完結し、画面の再構築は行わない。
- **配布形態(決定済み)**: 既存 CLI の新サブコマンド `cchub tui` として提供する(独立バイナリにはしない)。
- **セッション作成の範囲(v1)**: 新規作成はエージェント種別と作業ディレクトリの指定までを対象とする。初期プロンプト送信などの高度な作成オプションは将来拡張とする。
- **付随機能の扱い**: 会話ビュー・ファイル変更表示・ダッシュボード等は v1 のスコープ外(将来拡張)。v1 は一覧・入室・履歴検索・基本ライフサイクルに集中する。
- **実装方針(`/speckit-plan` で詳細化)**: 描画は TUI フレームワーク、データ取得は稼働中サーバの既存 HTTP/WS API の再利用、型は既存の共有型定義を流用、認証はローカルの既存資格情報を流用する方針が合意済み。具体の技術選定・API 対応・モジュール構成は計画フェーズで確定する。

## Dependencies

- 稼働中の CC Hub サーバ(セッション一覧・状態・履歴検索・セッション操作の供給元)。
- ローカルの tmux(セッション実体および入室=アタッチの実行基盤)。
